What is Docker?

Docker packages an application and its dependencies into a standardized unit called a container. Containers share the host OS kernel but run in isolated user-space, making them lightweight, fast, and reproducible across environments.

Installation

Docker Engine runs on Linux, macOS, and Windows (via WSL2).

Linux (Ubuntu/Debian)

curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
sudo usermod -aG docker $USER   # log out/in after this
docker --version

macOS / Windows

Install Docker Desktop from docker.com. It bundles the engine, CLI, Compose, and a lightweight VM.

Verify: run docker run hello-world — it pulls a tiny image and prints a success message.

Your First Container

# Run an interactive shell in a Ubuntu container
docker run -it --rm ubuntu:latest bash

# Run a detached nginx web server
docker run -d --name web -p 8080:80 nginx:latest
# open http://localhost:8080

docker ps                # list running containers
docker stop web          # stop it
docker rm web            # remove the stopped container
Flags explained: -d detached (background), -it interactive TTY, --rm auto-remove when stopped, -p HOST:CONT port mapping.

Working with Images

docker images                  # list local images
docker pull alpine:3.20        # download an image
docker rmi alpine:3.20         # remove an image
docker image prune -a          # remove all unused images
docker tag myapp:1.0 me/myapp:1.0   # create an alias tag

Writing a Dockerfile

A Dockerfile is a list of instructions. Each instruction creates a layer in the image.

# syntax=docker/dockerfile:1.7
FROM node:20-alpine           # base image
WORKDIR /app                  # cd into /app for subsequent commands
COPY package*.json ./         # copy dependency manifests first (better caching)
RUN npm ci --omit=dev         # install deps
COPY . .                      # copy source code
ENV NODE_ENV=production
EXPOSE 3000
CMD ["node", "server.js"]     # default command
docker build -t myapp:1.0 .          # build from current dir
docker run -p 3000:3000 myapp:1.0    # run it

Volumes & Data

Containers are ephemeral — data inside is lost when removed. Use volumes for persistence.

# Named volume (managed by Docker)
docker volume create db-data
docker run -d -v db-data:/var/lib/postgresql/data postgres:16

# Bind mount (a host directory mounted in)
docker run -d -v $(pwd)/src:/app/src myapp:1.0

docker volume ls
docker volume inspect db-data
Prefer named volumes over bind mounts for databases — they're easier to back up, migrate, and don't depend on host paths.

Networking Basics

docker network ls                       # list networks
docker network create mynet             # create a custom bridge
docker run -d --network mynet --name api myapi:1.0
docker run -d --network mynet -p 8080:80 --name web nginx   # web→api via DNS name "api"
docker network inspect mynet
DNS magic: on a user-defined network, containers resolve each other by name (here, api is reachable from web as http://api:80).

🟡 Intermediate

Docker Compose

Compose lets you declare multi-container apps in YAML and manage them as one.

# docker-compose.yml
services:
  web:
    build: .
    ports:
      - "8080:80"
    depends_on:
      - api
    environment:
      - API_URL=http://api:3000

  api:
    build: ./api
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/app
    depends_on:
      db:
        condition: service_healthy

  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data
    environment:
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
      - POSTGRES_DB=app
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d app"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:
  db-data:
docker compose up -d          # start
docker compose ps             # status
docker compose logs -f web    # stream logs
docker compose exec web sh    # shell into a service
docker compose down           # stop & remove containers/networks
docker compose down -v        # also remove volumes

Multi-stage Builds

Use multiple FROM stages to keep build tools out of the final image.

# syntax=docker/dockerfile:1.7
# ---- build stage ----
FROM golang:1.22 AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /out/app

# ---- runtime stage ----
FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /out/app /app
USER nonroot:nonroot
ENTRYPOINT ["/app"]
A Go binary from this Dockerfile is ~10–20 MB instead of ~800 MB — and has no shell, no package manager, and a non-root user by default.

Registries & Push/Pull

docker login                     # login to default registry (Docker Hub)
docker login ghcr.io             # GitHub Container Registry
docker tag myapp:1.0 ghcr.io/me/myapp:1.0
docker push ghcr.io/me/myapp:1.0
docker pull ghcr.io/me/myapp:1.0
docker logout
Don't bake secrets into images. Anything in a layer is in the image forever — even after you "delete" it in a later layer. Use env vars, secrets, or mount at runtime.

Environment & Configuration

# Pass env vars
docker run -e NODE_ENV=production -e LOG_LEVEL=info myapp:1.0

# From a file
docker run --env-file .env myapp:1.0

# Use BuildKit secrets at build time (never stored in image)
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci

Healthchecks

HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD curl -fsS http://localhost:3000/health || exit 1

docker ps shows the (healthy)/(unhealthy) status. Compose can gate service startup on condition: service_healthy.

🔴 Advanced

Security Hardening

  1. Run as non-root. Add USER in your Dockerfile.
  2. Minimal base images. Prefer alpine, distroless, or scratch.
  3. Don't bake secrets. Use BuildKit secrets, env vars at runtime, or a secret manager.
  4. Pin versions. Avoid latest in production — pin by digest for reproducibility.
  5. Read-only filesystems. docker run --read-only --tmpfs /tmp myapp
  6. Drop capabilities. --cap-drop=ALL --cap-add=NET_BIND_SERVICE
  7. Scan images. docker scout cves myapp:1.0 or trivy image myapp:1.0.
docker run --rm \
  --read-only \
  --tmpfs /tmp \
  --cap-drop=ALL \
  --security-opt no-new-privileges \
  --user 1000:1000 \
  myapp:1.0

Image Size & Performance

  • Order Dockerfile instructions from least to most frequently changing (leverage layer cache).
  • Combine RUN steps; clean package manager caches in the same layer.
  • Use .dockerignore to exclude node_modules, .git, build artifacts.
  • Multi-stage builds to throw away build toolchain.
  • Use slim/alpine base images when feasible.
# Good: single layer, cached
RUN apt-get update && apt-get install -y --no-install-recommends \
        curl ca-certificates \
    && rm -rf /var/lib/apt/lists/*

Debugging & Inspection

docker logs -f --tail 100 web           # follow logs
docker exec -it web sh                 # shell into a running container
docker inspect web                     # full JSON config
docker stats                           # live CPU/mem/net
docker top web                         # processes inside
docker diff web                        # filesystem changes vs image
docker events --since 10m              # daemon events
docker system df                       # disk usage
docker system prune -a                 # nuke everything unused

For a network-level view: docker run --rm -it --network container:web nicolaka/netshoot gives you curl, nslookup, tcpdump, etc.

BuildKit & Cache

BuildKit is the modern builder: parallel builds, better caching, secrets, mounts.

# Enable
export DOCKER_BUILDKIT=1

# Build with a remote cache (registry)
docker buildx build \
  --tag ghcr.io/me/myapp:1.0 \
  --cache-from type=registry,ref=ghcr.io/me/myapp:buildcache \
  --cache-to   type=registry,ref=ghcr.io/me/myapp:buildcache,mode=max \
  --push .

# Mount cache for package managers (huge speedup)
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt
Use --mount=type=cache for pip, npm, go mod, cargo — dependencies survive across builds without bloating the image.

Advanced Networking

  • bridge (default): isolated, NAT to host.
  • host: no isolation, container shares host network stack. Fastest, least safe.
  • none: no networking — bring your own.
  • overlay: multi-host networking (Swarm/Kubernetes).
  • macvlan: container appears as a physical device on your LAN.
# Run with host networking (only when you need max performance and trust the workload)
docker run --network host nginx

# Attach a second network to a running container
docker network connect monitoring web

⚡ Pro Tips

Pin by digest for reproducibility. Tags are mutable; digests aren't. Use node:20-alpine@sha256:<digest> in production.
One process per container. Compose for multi-service apps. A container doing five things is hard to scale, debug, and update.
Log to stdout/stderr. Let the Docker logging driver handle aggregation — don't write to files inside the container (they're lost without explicit mounts).
Use exec form for CMD/ENTRYPOINT. CMD ["node","server.js"] doesn't shell-wrap, so signals (SIGTERM) reach your app for graceful shutdown.
Set a stop grace period. docker run --stop-timeout 30 ... or in Compose stop_grace_period: 30s. Containers that ignore SIGTERM hang the orchestrator.
Build with --pull in CI. docker build --pull -t myapp:$CI_SHA . ensures you get the latest base image, not a stale cache.
Use BuildKit secrets and ssh mounts. Never COPY a token into a layer; mount it at build time and it's not in the image.
Combine HEALTHCHECK with Compose depends_on: condition: service_healthy so dependent services wait for real readiness, not just "process started."
Use docker init. Docker 24+ scaffolds a sensible .dockerignore, Dockerfile, and Compose file for Node/Python/Go/Java. Great starting point.
Image scanning in CI. Block merges on HIGH/CRITICAL CVEs: trivy image --exit-code 1 --severity HIGH,CRITICAL myapp:$SHA.
Smaller is safer. Every extra binary is an extra attack surface and bug surface. distroless/scratch for Go, alpine for Node/Python, slim for Java.
Use docker compose --profile dev up to keep dev-only services (adminer, mailhog) out of production.
Don't docker system prune -a on a shared host. It removes all stopped containers and unused images — including other people's. Scope it or use --filter.
Resource limits. Always set --memory and --cpus in production; one runaway container can starve the host.
